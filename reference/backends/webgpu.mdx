The LLogos WebGPU backend (`@daisinet/llogos-webgpu`) runs GGUF model inference in any WebGPU-capable browser or Node.js environment, using 20+ WGSL compute shaders for GPU-accelerated text generation.

## Installation

```bash
npm install @daisinet/llogos-webgpu
```

## Architecture

The WebGPU backend is written in TypeScript and runs entirely in the browser or in Node.js via Dawn bindings. It has no dependency on the .NET backends.

```
Browser Tab / Node.js
┌───────────────────────────────────────────┐
│  LlogosEngine                             │
│  ┌─────────────────────────────────────┐  │
│  │ GGUF Parser → Tensor Loader → Model │  │
│  │ BPE Tokenizer (from GGUF metadata)  │  │
│  │ Sampler (temperature, top-k, top-p) │  │
│  └─────────────────────────────────────┘  │
│                    │                      │
│  ┌─────────────────────────────────────┐  │
│  │ ComputeEngine                       │  │
│  │  BufferPool      ShaderCache        │  │
│  │  (VRAM tracking) (pipeline cache)   │  │
│  │  20+ WGSL compute shaders           │  │
│  └─────────────────────────────────────┘  │
│                    │                      │
│           WebGPU API (navigator.gpu)      │
└───────────────────────────────────────────┘
                     │
             GPU Hardware
     (NVIDIA / AMD / Intel / Apple)
```

Four primary classes compose the engine:

| Class | Responsibility |
|-------|---------------|
| `LlogosEngine` | Top-level API — GPU init, model loading, token generation, sampling |
| `ComputeEngine` | Dispatches all GPU operations, manages shader pipelines and barriers |
| `BufferPool` | Allocates and reuses GPU buffers, tracks VRAM usage |
| `ShaderCache` | Caches compiled `GPUComputePipeline` objects to avoid redundant compilation |

## WGSL compute shaders

All GPU computation is expressed in WGSL (WebGPU Shading Language). Each shader is compiled to a `GPUComputePipeline` on first use and cached by `ShaderCache`.

| Shader | Operation | Workgroup |
|--------|-----------|-----------|
| `matmul.wgsl` | F32 matrix-vector multiply | 256 threads, 2D dispatch for large M |
| `matmul_q4.wgsl` | Fused Q4_0 dequant + matmul | 256 threads |
| `matmul_q8.wgsl` | Native Q8_0 matmul (`unpack2x16float`) | 256 threads |
| `attention.wgsl` | Multi-head attention with GQA | 64 threads per head |
| `rmsnorm.wgsl` | RMS normalization | 256-thread reduction |
| `rope.wgsl` | Rotary position embeddings | 1 thread per pair |
| `embedding.wgsl` | Token embedding lookup | 256 threads |
| `silu_mul.wgsl` | Fused SiLU gate × up (SwiGLU) | 256 threads |
| `add.wgsl` | Element-wise add (residual connections) | 256 threads |
| `add_bias.wgsl` | In-place bias addition | 256 threads |
| `copy_rmsnorm.wgsl` | Fused copy + RMSNorm | 256 threads |
| `softmax.wgsl` | Numerically stable softmax | 256 threads |
| `conv1d_silu.wgsl` | Fused causal conv1d + SiLU with persistent state | 256 threads |
| `l2_norm_groups.wgsl` | Per-group L2 normalization | 256-thread reduction per group |
| `compute_decay_beta.wgsl` | Softplus + exp decay, sigmoid beta | 64 threads |
| `deltanet_step.wgsl` | Full state update: mat-vec, rank-1 update, output, per-head RMSNorm | 128 threads per group |
| `silu_gate.wgsl` | Element-wise SiLU gating | 256 threads |
| `silu_inplace.wgsl` | In-place SiLU activation | 256 threads |

The 6 DeltaNet-specific shaders (`conv1d_silu`, `l2_norm_groups`, `compute_decay_beta`, `deltanet_step`, `silu_gate`, `silu_inplace`) run entirely on GPU with zero CPU readbacks during inference.

## HTTP model loading and caching

Models are loaded via HTTP `fetch` with streaming progress. Once downloaded, the raw GGUF bytes are stored in the browser's **Cache API** — subsequent page loads skip the network entirely and serve the model from the cache.

```typescript
// First load: downloads from URL and stores in Cache API
await engine.loadModel('https://huggingface.co/.../model.gguf');

// Subsequent loads: served from Cache API (no network request)
await engine.loadModel('https://huggingface.co/.../model.gguf');
```

The `DownloadManager` class in `src/storage/download-manager.ts` handles both paths transparently.

## Chat template engine

The engine automatically detects and applies the correct chat template for each model:

| Template | Detection | Format |
|----------|-----------|--------|
| **Llama 3** | `<\|start_header_id\|>` token in vocab | `<\|begin_of_text\|>` + role headers |
| **ChatML** | `<\|im_start\|>` token in vocab | `<\|im_start\|>role\ncontent<\|im_end\|>` |
| **Llama 2** | `[INST]` pattern in `tokenizer.chat_template` | Simple instruction wrapping |
| **Jinja2** | Generic parser for `tokenizer.chat_template` | `{% for %}`, `{% if %}`, `{{ var }}`, filters |

The Jinja2 parser reads the `tokenizer.chat_template` field from GGUF metadata and evaluates it directly — no server-side rendering required.

## DAISI network integration

The WebGPU engine integrates with the DAISI Manager as a **Browser Host**:

- Loads models from HuggingFace via HTTP, persists in browser Cache API
- Connects to the Orchestrator (ORC) directly via gRPC-web
- Processes inference requests from the DAISI network
- Streams tokens back in batches of 10 per response for efficient throughput
- Auto-reconnects with exponential backoff on connection loss

## TypeScript API

```typescript
import { LlogosEngine } from '@daisinet/llogos-webgpu';

const engine = new LlogosEngine();

// Initialize the WebGPU device
await engine.initGpu();

// Load a GGUF model (cached after first download)
await engine.loadModel('https://huggingface.co/.../model.gguf');

// Generate tokens as an async iterable
for await (const token of engine.generate('Hello')) {
  process.stdout.write(token);
}
```

## Browser and runtime requirements

<Note>
WebGPU is available in Chrome 113+, Edge 113+, and Safari 18+ (with the WebGPU feature flag). Firefox support is in progress behind a flag.
</Note>

| Environment | Requirement |
|------------|-------------|
| Chrome | 113+ |
| Edge | 113+ |
| Node.js | Via `webgpu` npm package (Dawn WebGPU bindings) |
| Android | Tested (Adreno / Mali via WebGPU) |
| iOS | Not yet supported |

Node.js support uses the `webgpu` npm package, which bundles Google's Dawn native WebGPU implementation. This enables automated CI testing without a browser.
