LLogos includes a built-in benchmark mode and an interactive Next.js dashboard for comparing performance across models, backends, and KV strategies.

The `--bench` flag measures prefill and decode throughput separately — prefill processes all prompt tokens in parallel (compute-bound), while decode generates one token at a time (memory-bandwidth-bound). Reporting them separately gives a complete picture of where throughput is constrained.

<Tabs>
  <Tab title="CLI benchmark">
    ## Running benchmarks from the CLI

    Add `--bench` to any inference command. LLogos runs the model, times both phases independently, and prints a results table:

    ```bash
    dotnet run --project src/Daisi.Llogos.Cli -- \
        --model C:\GGUFS\Qwen3.5-0.8B-Q8_0.gguf \
        --bench --backend cuda
    ```

    Output:

    ```
    === Benchmark Results ===
      Prefill:     128 tokens in      42.3 ms  (3026.0 tok/s)
      Decode:      128 tokens in     290.1 ms  ( 441.2 tok/s)
      Total:       256 tokens in     332.4 ms
      Load:                          1241.0 ms
    ```

    The default benchmark prompt is `"The meaning of life is"`. Pass `--prompt` to use a custom prompt, or vary `--max-tokens` to control decode length.

    Benchmark with a specific KV strategy:

    ```bash
    # Benchmark with LLogos Turbo KV compression
    dotnet run --project src/Daisi.Llogos.Cli -- \
        --model C:\GGUFS\Qwen3.5-9B-Q8_0.gguf \
        --bench --backend cuda --kv-quant turbo:3
    ```

    Benchmark on Vulkan (AMD, Intel, or NVIDIA via cross-platform GPU):

    ```bash
    dotnet run --project src/Daisi.Llogos.Cli -- \
        --model C:\GGUFS\Qwen3.5-0.8B-Q8_0.gguf \
        --bench --backend vulkan
    ```

  <Tab title="Bench dashboard">
    ## LLogos Bench dashboard

    LLogos Bench is an interactive Next.js dashboard for visual benchmark comparison across models, backends, and KV strategies.

    ```bash
    cd src/bench
    npm install
    npm run dev   # http://localhost:3000
    ```

    The dashboard auto-discovers GGUF model files, runs benchmarks by invoking the CLI, and streams results cell-by-cell as they complete.

    ### What the dashboard shows

    - **System specs header** — CPU, GPU, VRAM, RAM, OS, .NET version
    - **Model selector** grouped by architecture (DeltaNet Hybrid, Standard Attention, BitNet) with select/deselect-all per group
    - **Configuration matrix** — F16 baseline, Turbo 2/3/4-bit, QJL variants
    - **Backend toggle** — CPU or CUDA
    - **Context length toggle** — Short (~150 tokens) or Long (~4K tokens)
    - **Color-coded cells** relative to the matched baseline
    - **Compression ratio display** for Turbo configurations
    - **Live SSE streaming** with cancel support — results appear as they finish, not all at once

    Each cell shows prefill tok/s and decode tok/s for the selected model, backend, and configuration combination.

## WebGPU benchmarks

LLogos also runs in the browser via WebGPU. Benchmarks were measured using the Dawn WebGPU runtime in Node.js on an NVIDIA RTX 5090 (Blackwell), 32 decode tokens:

| Model | Prefill | Decode | VRAM |
|-------|--------:|-------:|-----:|
| TinyLlama 1.1B Q8_0 | 45 tok/s | — | 1570 MB |
| Llama 3.2 1B Q8_0 | 61 tok/s | 54 tok/s | 2787 MB |
| Qwen 2.5 0.5B Q8_0 | 42 tok/s | 37 tok/s | 1592 MB |
| Qwen 3.5 0.8B Q8_0 | 17 tok/s | 17 tok/s | 1592 MB |

Qwen 3.5's DeltaNet layers run entirely on GPU via 6 custom WGSL shaders with zero CPU readbacks. The lower tok/s for Qwen 3.5 vs Qwen 2.5 at similar model sizes reflects the additional DeltaNet computation (state-space update + causal conv1d) that does not exist in standard attention models.

To run WebGPU benchmarks:

```bash
cd src/webgpu
npm install
npx vitest run test/benchmark.test.ts
```

## Understanding prefill vs decode

The `--bench` output separates two distinct phases:

| Phase | What it measures | Bottleneck | Scales with |
|-------|-----------------|-----------|-------------|
| **Prefill** | Processing all prompt tokens in parallel | GPU compute (TFLOPS) | Prompt length, model size |
| **Decode** | Generating each output token one at a time | Memory bandwidth (GB/s) | Context length, KV cache size |

Prefill throughput is compute-bound: a longer prompt gives more parallelism and typically higher tok/s. Decode throughput is memory-bound: each step reads all model weights and KV cache from VRAM, so it is primarily limited by memory bandwidth rather than compute.

For latency-sensitive applications, prefill tok/s determines time-to-first-token. For throughput-sensitive applications (long generations), decode tok/s is what matters.
